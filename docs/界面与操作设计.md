# 界面与操作设计

这份文档用于约束 Lan Control Hub 的桌面端和手机端体验。目标是让普通用户不用理解协议、端口和能力字段，也能完成加入房间、找设备、聊天、传文件、看屏、远控和手机快捷操作。

## 设计原则

1. 默认界面只放日常操作：设备、聊天、文件、终端、看屏、远控、设置。
2. 高级能力默认收起：Local API、CLI、Agent、诊断、防火墙修复、WebRTC 中继配置。
3. 所有危险操作要有明确边界：信任设备、退出房间、远控、终端和防火墙修复都不能静默执行。
4. 文案使用用户能理解的词：房间、房间密码、设备代号、手机地址、快捷动作；避免直接展示内部 capability 名称。
5. 每个页面只回答一个主问题：我在哪个房间、当前走哪个网络入口、有哪些设备、我能对它做什么、出问题去哪里修。

## 桌面端信息架构

### 首次启动

首次启动只提供两个选择：

- 创建新房间：当前还没有要加入的 LCH 信任圈，或用户想重新开始。
- 加入已有房间：同局域网先扫描附近入口，再输入房间密码；跨网时直接粘贴房间密钥。

扫描结果要像“附近入口列表”，显示房间名称、房间代码、局域网入口地址和在线设备数量。扫描不到时，引导用户创建新房间，或说明跨网需要 Tailscale 100.x 入口。

### 主工作台

主工作台负责日常设备操作：

- 左侧：搜索、在线设备、房间/分组。
- 中间：当前选中设备、常用操作、设备列表。
- 右侧：设备别名、房间、收藏、只读、最近任务、远控会话。

设备卡片只显示用户能理解的能力标签，例如“聊天”“文件”“终端”“看屏”“远控”“剪贴板”。不要直接显示 `terminal.pty`、`chat.markdown` 这类内部能力名。

### 设置页

设置页按使用频率分层：

- 网络：房间状态、加入新设备、退出房间、局域网入口、Tailscale/ZeroTier/WireGuard 入口、WebRTC 连接。
- 信任：待信任设备、可信设备、自动信任策略。
- 文件共享：共享目录和本机共享状态。
- 系统：版本更新、手机控制台、高级工具。

系统页默认先给用户看“版本更新”和“手机控制台”。Local API、CLI、Agent、诊断、防火墙修复统一放进“高级工具”，默认折叠，避免普通用户一打开设置就看到太多技术项。

## 手机控制台

手机控制台不能照搬桌面端布局。它应该是手机优先的信息架构，底部页签固定为：

- 总览：房间、网关、在线数量、快捷动作和语音入口。
- 控制：选择 CLI 可用设备，使用聊天窗口与该设备的智能体对话。
- 设备：查看设备在线、信任、只读和 CLI 配置状态。
- 任务：查看最近输出。

手机端所有按钮必须有清晰文字，不能只显示图标。语音只填入输入框，不自动执行；高危动作要在执行前确认。

同局域网使用时，手机打开：

```text
http://电脑IP:46882/mobile/
```

跨网络使用时，推荐手机和其中一台电脑同时加入 Tailscale、ZeroTier 或 WireGuard，再访问这台电脑的虚拟专网地址。其它局域网电脑由这台 LCH 节点通过应用层转发，不要求公网端口暴露。

手机端的“控制”页用于实现这条路径：

```text
手机选择设备 -> 输入或语音填入 -> 已配置的 CLI 智能体回复到聊天窗口
```

设备目标必须明确区分：

- CLI 可用：允许发送到 Claude Code / MiniMax-M3 这类已配置智能体。
- CLI 未配置：只显示状态，不允许发送自然语言命令。
- 离线、未信任、只读：不能作为控制目标。

## API 和 Agent 设置

Local API 是给本机脚本、CLI 和 Agent 使用的，不是普通用户的主入口。界面要求：

- 默认只显示“高级工具”折叠入口。
- 展开后显示本机监听地址和常用命令。
- 明确写出 Local API 只监听 `127.0.0.1`。
- 不把公网访问作为默认方案。

后续如果接入语音命令或快捷指令，建议优先走手机控制台和 Local API 的白名单动作，不直接暴露任意 shell。

## 出错时的处理路径

用户遇到问题时，界面应优先给出最短处理路径：

- 看不到设备：检查同一房间、同一局域网或虚拟专网、等待 5 到 10 秒、刷新。
- 不能控制：确认双方互相信任、目标设备在线、防火墙未阻止。
- 实时画面不稳定：先保持直连优先；跨网络再配置 STUN/TURN。
- 加错房间：设置页网络里退出当前房间，再重新扫描加入。
- 手机打不开：确认手机和电脑在同一局域网或 Tailscale，复制系统页里的手机地址。

## 后续 UI 检查清单

每次迭代桌面端 UI 时检查：

- 新功能是否放在正确页面，而不是全塞进设置或弹窗。
- 普通用户默认是否只看到必要操作。
- 技术项是否有折叠、说明和清晰边界。
- 设备名、房间名、设备代号和网络入口是否分开展示，不混用内部 ID 或把 192/100 地址说成两个房间。
- 小窗口和窄屏下文字是否换行，不横向溢出。
- 旧设备不支持的新能力是否降级显示，而不是报错。
